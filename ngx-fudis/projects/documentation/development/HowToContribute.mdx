# How to Contribute

Fudis Design System is open for new development contributions in - and outside of Funidata company.

## Before contributing

**All Fudis Design System proposals must be generic and they must solve commonly repeated challenges across learning services.** If the proposal is considered as project spesific, it will be discarded.
Therefore, it's important to define the scope of your proposal â€” whether it's an improvement to an existing component or a completely new feature. The best way to get started is by becoming familiar with the existing Fudis component library.
In addition to new feature proposals, contributions may include bug fixes or just documentation update requests. Do not hesitate to contact Fudis team in advance for help and guidance.

## Contribution guidelines (external developer):

- Define your implementation proposal.
- [Contact Fudis Team](/docs/documentation-introduction-welcome--documentation#contact) in order to find best way contribute.
  - **Note!** New features require a design that Fudis team will provide consultancy on.
- **Implementation step-by-step**
  1. Get familiar with the Fudis development practices as documented in [DS Practices](/docs/documentation-development-ds-practises-ways-of-working--documentation).
  2. [Fork the repository](https://github.com/funidata/fudis/fork). Select yourself as the owner and press "Create fork".
  3. Set up your local development environment as documented in [Getting Started](/docs/documentation-development-getting-started--documentation).
  4. Create a new branch, as guided in [Git and Github Convension](/docs/documentation-development-tooling-git-and-github-conventions--documentation).
  5. Create your implementation including the following:
     - Code and Design ( Main styles live in Core Style Library )
     - Unit tests and Visual regression tests
     - Documentation
  6. Make a pull request on [Fudis Github page](https://github.com/funidata/fudis).
  7. Fudis team will review the PR, merge it or request for new changes. All open PRs are reviewed to satisfy the following acceptance criteria: accessibility, usability, security, modularity and performance.

## Contribution guidelines (Funidata developer):

- [Contact Fudis Team](/docs/documentation-introduction-welcome--documentation#contact) and propose your implementation.
- Together with the Fudis team, will elaborate who is responsible for development.
- Implementation ticket is created on DS Backlog and prioritized for a future release.
- When the proposal design changes are available, implementation ticket is being addressed to contributing developer.
